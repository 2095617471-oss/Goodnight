# 往日回音日历渲染数据源修改计划

## 1. 目标与需求分析
当前“往日回音”页面中的心情日历是依据日记的**创建时间**（即用户点击“封存日记”时的时间）来展示记录的。根据用户需求，需要将日历的渲染依据修改为**事件实际发生时间**（即用户在向导第一步中选择的年、月、日），同时：
- 保留原本的“创建时间/提交时间”字段。
- 确保事件列表等原有功能不受影响，数据和界面能正常联动。
- 处理好旧数据的兼容性。

## 2. 现状分析
通过对 `js/app.js` 代码的分析，当前系统的运作机制如下：
- **数据结构**：日记保存在 localStorage 中，对象包含 `date` 字段（记录的是创建时的 `ISOString`），以及通过向导收集的 `wizardContext`（包含 `time`、`location`、`people`、`action`）。
- **时间数据**：目前向导第一步的实际发生时间被拼接成了一段文本，例如 `"2026年5月3日 晚上"`，存储在 `wizardContext.time` 中。
- **日历渲染**：`renderCalendar()` 方法中通过 `new Date(d.date)` 获取日期对象，并与当前日历面板的年月进行比对。
- **界面联动**：日历格子点击后，会通过 `d.id` 寻找对应的日记卡片进行锚点跳转。由于 ID 不会改变，因此联动逻辑天然不受时间字段修改的影响。

## 3. 具体修改方案

### 3.1 修改向导数据收集逻辑（写入阶段）
为了更规范地存储“实际发生时间”，我们将在 `wizardContext` 中新增一个结构化的 `eventDate` 字段（格式为 `YYYY-MM-DD`）。
- **涉及文件**：`js/app.js`
- **操作**：
  1. 在 `state` 初始定义和 `resetWizard` 函数中，为 `wizardData` 对象补充 `eventDate: ''` 属性。
  2. 在 `collectWizardData` 函数中，处理 `wizard-step-1` 时，除保留原有的 `time` 字符串拼接外，额外获取选择的年、月、日，格式化后赋值给 `state.wizardData.eventDate`。

### 3.2 修改日历渲染逻辑（读取阶段）
将日历格子渲染所依赖的时间数据源从“创建时间”切换为“实际发生时间”，并添加对旧数据的兼容处理。
- **涉及文件**：`js/app.js`
- **操作**：
  在 `renderCalendar()` 函数遍历 `diaries` 的循环中，修改日期对象的获取逻辑：
  1. 优先尝试读取 `d.wizardContext.eventDate`（新数据格式），如果存在则解析该日期。
  2. 如果不存在（说明是历史遗留数据），则尝试使用正则表达式 `/(\d+)年(\d+)月(\d+)日/` 从 `d.wizardContext.time` 字符串中提取出年、月、日并生成日期对象。
  3. 如果均未提取成功，则回退使用原有的 `new Date(d.date)`。
  4. 使用提取到的实际发生日期对象（`eventDateObj`）去判断是否属于当前日历展示的月份，并决定它落在哪个日期格子上。

### 3.3 保持原有功能不受影响
- **事件列表**：`renderDiaryList()` 函数中继续使用 `d.date` 作为显示的时间，这样列表中显示的仍然是日记的“创建/记录时间”。
- **跳转联动**：由于日历与列表的联动完全依赖日记对象的唯一标识符 `id`，只要不修改 `id` 属性，点击日历格子依然能准确跳转到对应的日记卡片。

## 4. 验证与测试
- 记录一篇带有过去时间（例如“5月1日”）的新日记，验证日历上是否在“5月1日”展示情绪格子。
- 验证日记列表是否正常显示该日记，且显示的时间为当前的真实时间。
- 点击日历上的“5月1日”格子，验证是否能平滑滚动到刚创建的日记卡片。
- 确认之前创建的旧日记在日历中依然能够正确显示（通过正则兼容提取的时间）。
